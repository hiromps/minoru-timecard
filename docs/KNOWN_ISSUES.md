# ミノルタイムカードシステム 既知の課題・注意点

このドキュメントは、ミノルタイムカードシステムの運用・保守にあたって把握しておくべき既知の課題と注意点を
まとめたものです。各項目には「**現状／影響／対応方針**」を記載しています。

対象読者は、システムの保守・運用を行う担当者と管理者（人事担当）です。
日常の操作手順は [USER_GUIDE.md](./USER_GUIDE.md)（従業員向け）／[ADMIN_GUIDE.md](./ADMIN_GUIDE.md)（管理者向け）を参照してください。

## 1. ステータス再計算・タイムゾーンに関する注意

**現状**
- 打刻時刻はデータベースに `timestamptz`（タイムゾーン付き）型でUTC基準で保存され、画面には日本時間（JST）で表示されます。
- 過去に、サーバー（ホスト環境）のタイムゾーンがUTCの状況で、時刻の解釈がずれて**約9時間のズレ**が発生する不具合がありました。
  特に JST の午前0時〜9時台の打刻で、日付が前日にずれ、退勤時刻が出勤時刻より前の絶対時刻になり、
  データベース制約（退勤 > 出勤）に違反して退勤打刻ができない、という症状が出ていました。
- 調査の履歴として、次の診断用SQLが `claudedocs/` に残っています（調査記録であり、日常運用では使用しません）。
  - `claudedocs/diagnose_status.sql`: 列型（`timestamptz` か `timestamp` か）の確認、保存値のUTC/JST両解釈での突き合わせ、
    正しいロジックで再計算したステータスとの「答え合わせ」、不一致件数の集計。
  - `claudedocs/check_rls.sql`: ステータス再計算が反映されない原因の切り分け（RLSポリシー・テーブル権限の確認）。

**影響**
- 時刻の型（`timestamp without time zone` を誤って使う）やホスト環境のタイムゾーン設定を誤ると、
  打刻時刻・ステータスが実態とずれるリスクがあります。
- 表示上は正しく見えても、集計・給与計算の元データがずれる可能性があります。

**対応方針（修正済 + 運用回避）**
- 現在は、打刻・修正の時刻生成をJST基準に統一する処理（JST基準の日付取得と、その逆関数による相互変換）で修正済みです。
- 時刻の型は `timestamptz` を前提とします。データベースのスキーマ変更時は型を変えないでください。
- 社員の始業・終業時刻を変更した後などにステータスが実態と合わない場合は、管理者画面の
  「**ステータス再計算**」（[ADMIN_GUIDE.md](./ADMIN_GUIDE.md) 第4章）を実行してください。
- それでも解消しない場合は、上記の診断SQLを参考に、列型・タイムゾーン・RLS権限を確認します。

## 2. `isDevMode`（デモモード判定）が2箇所で定義されている

**現状**
- デモモードかどうかを判定する `isDevMode` が、`src/lib/supabase.ts` と `src/lib/mockData.ts` の**2箇所で個別に定義**されています。
- どちらも `REACT_APP_SUPABASE_URL` 環境変数の有無などから判定しており、条件は同等ですが、定義が重複しています。
  現状、認証系（`src/lib/auth.ts`）は `mockData.ts` の定義を、その他の多くのモジュールは `supabase.ts` の定義を参照しています。

**影響**
- 判定条件を変更する際に片方だけ直すと、モジュール間で挙動が食い違うリスクがあります（保守上の注意点）。
- 現時点で実運用上の不具合は確認されていません。

**対応方針（解消済み）**
- `isDevMode` の定義を `src/lib/supabase.ts` に一本化し、`src/lib/mockData.ts` は同モジュールから import して再エクスポートする形へ統一済みです（循環参照なし）。
- 今後、判定ロジックを変更する場合は `src/lib/supabase.ts` の定義のみを更新してください。

## 3. `.env.example` に宣言のみで未使用の環境変数がある

**現状**
- `.env.example` には、宣言されているものの**現在のコードでは参照されていない（機能として未実装の）**環境変数が含まれています。
  - `REACT_APP_PRODUCTION`
  - `REACT_APP_MAX_LOGIN_ATTEMPTS`
  - `REACT_APP_LOCKOUT_DURATION`
  - `REACT_APP_SESSION_DURATION`
  - `REACT_APP_AUDIT_LOG_ENABLED`
  - `REACT_APP_SESSION_MANAGEMENT_ENABLED`
  - `REACT_APP_FAILED_LOGIN_PROTECTION`

**影響**
- これらを設定しても、対応する機能（ログイン試行回数制限、ロックアウト、セッション有効期限の強制、監査ログ有効化フラグ等）は
  **自動的には有効になりません**。「設定したから安全」という誤解を招く恐れがあります。
- セキュリティ機能が有効であるかのように見えるため、運用上の過信につながる点に注意が必要です。

**対応方針（記録のみ）**
- 現状は「宣言のみ・未実装」であることを本ドキュメントに記録します。実際のセキュリティ設定・前提は
  [SECURITY.md](./SECURITY.md) を参照してください。
- 将来これらの機能を実装する場合は、変数名を流用しつつ実装を追加します。実装しない場合は `.env.example` から
  削除するか、コメントで「未実装」であることを明記することを推奨します。

## 4. 公開RLS（誰でも打刻できる前提）

**現状**
- 打刻画面（`/`）は、社員が個別ログインをせずに社員を選んで打刻する運用です。これを成立させるため、
  打刻データへのアクセスは**公開的なRLS（Row Level Security）ポリシー**を前提としています。
  すなわち、打刻画面にアクセスできる人は、匿名（未ログイン）でも打刻データの登録・参照が可能です。

**影響**
- 打刻画面のURLにアクセスできる環境からは、誰でも任意の社員として打刻できます（なりすまし打刻の余地があります）。
- アクセス経路（ネットワーク）そのものを制限しないと、データ保護のレイヤーが弱くなります。

**対応方針（運用回避）**
- 本システムは**社内ネットワーク内での利用を前提**とし、ネットワーク／IPレベルのアクセス制限で保護する設計です。
  外部に公開しないでください。
- なりすまし打刻の防止は運用ルール（本人による打刻の徹底、管理者による記録確認）で補います。
- アクセス制限の具体的な設定は [SECURITY.md](./SECURITY.md) を参照してください。

## 5. レガシー `backend/`（廃止済み・未デプロイ）

**現状**
- リポジトリには旧構成の `backend/`（Node.js + Express + SQLite）ディレクトリが残っていますが、
  現行システムでは**使用されていません（廃止／retired）**。現行はフロントエンド + Supabase 構成で動作します。
- このレガシーコードには、JWTの署名鍵の既定値が弱い箇所があります
  （`backend/src/middleware/auth.ts` と `backend/src/controllers/adminController.ts` で
  `JWT_SECRET` が未設定時に `'your-secret-key'` にフォールバック）。

**影響**
- 弱い既定の署名鍵は、本来であればトークン偽造のリスクになります。
- ただし、この `backend/` は**デプロイ・起動されていない**ため、現行の稼働システムに対する**実害はありません**。

**対応方針（記録のみ）**
- 現時点では「未使用・未デプロイのため実害なし」として記録します。
- 将来このレガシーバックエンドを削除する、または誤って復活・デプロイしないよう注意します。
  仮に再利用する場合は、`JWT_SECRET` を必ず環境変数で強固な値に設定し、弱い既定値のフォールバックを削除してください。

## 6. 監査ログ（`audit_logs`）のカラム系統が2つに分かれている

**現状**
- 監査ログへの書き込み経路が2系統あり、使用カラム名が一致していません。
  - データベースのトリガー関数 `audit_trigger_function` は `old_data` / `new_data` / `user_id` に書き込みます。
  - フロントエンドの記録処理（`src/lib/security.ts` の監査記録）は `new_values`（および `record_id` / `reason` / `ip_address` / `user_agent`）に書き込みます。
- 実データベース上で両系統のカラムが実在するのか、片方のみなのかは未確認です（`docs/DATA_MODEL.md` の「既知の不整合・推定箇所」に対応）。

**影響**
- 監査ログを集計・検索する際、どちらのカラムに値が入っているかを意識しないと、変更前後の値が取得できないことがあります。
- 一方のカラムしか存在しない場合、もう一方の経路の書き込みが失敗している可能性があります。

**実測で判明した事実（2026-07-06 に本番DBをMCPで確認）**
- 実 `audit_logs` テーブルの列は **`old_values` / `new_values`（jsonb）** です（`old_data` / `new_data` は存在しません）。
- フロントの記録処理は `new_values` に書き込むため**正しく動作**しています（実データ4件もこの経路由来）。
- 一方、DB関数 `audit_trigger_function` は存在しない `old_data` / `new_data` にINSERTしようとしており**壊れています**。ただし**どのテーブルにもトリガーとして設置されていない**（トリガー0件）ため発火せず、実害はありません（デッドコード）。

**対応（2026-07-06 実施）**
- `audit_trigger_function` の列名を `old_values` / `new_values` に修正済み（`record_id` も補完）。マイグレーション `supabase/migrations/0008_fix_audit_trigger_function_columns.sql`。
- ただしトリガーは意図的に**未設置**のままです（打刻ごとに監査行が増えるのを避けるため）。DB トリガーで自動監査を有効化したい場合は、修正済み関数を各テーブルに `CREATE TRIGGER` で紐付けてください。
- 監査ログは引き続き `old_values` / `new_values` を参照してください。

## 7. クライアント側のIP制限（`REACT_APP_ALLOWED_IPS`）は実効的な遮断にならない

**現状**
- `src/lib/security.ts` は `REACT_APP_ALLOWED_IPS` を参照するIP制限ロジックを持ちますが、クライアント（ブラウザ）側で動作するため、
  クライアントIPを取得する `getClientIP()` はプレースホルダ（`0.0.0.0` 相当）を返します。
- そのため、`REACT_APP_ALLOWED_IPS` を設定しても、ブラウザ側での実効的なアクセス遮断にはなりません。

**影響**
- 「アプリ側でIP制限しているから安全」という誤解につながります。実際にはブラウザ内のチェックは回避可能であり、防御境界として機能しません。
- 公開RLS（項目4）と併せ、ネットワーク層で制限しない限りアクセス保護は弱いままです。

**対応方針（記録のみ・運用回避）**
- アクセス制限は**アプリではなくネットワーク／ホスティング層**で行ってください（社内ネットワーク限定、リバースプロキシ/ファイアウォールでのIP制限、Vercel の Deployment Protection 等）。詳細は [SECURITY.md](./SECURITY.md)。
- `REACT_APP_ALLOWED_IPS` を実効的なセキュリティ対策とみなさないでください。

## 8. 本番モードで環境変数未設定時に「静かにデモモードへフォールバック」する

**現状**
- `src/lib/supabase.ts` は、本番モード（`isDevMode=false`）でも `REACT_APP_SUPABASE_URL` / `REACT_APP_SUPABASE_ANON_KEY` が未設定・無効な場合、
  クライアント生成時の例外を `try/catch` が捕捉し、最終的にモッククライアント（デモ相当）へフォールバックします。
- WS-Eの修正で、URL/anonキーが無効な場合は明確な例外を投げるようになりましたが、この例外は既存の `try/catch` に捕捉されるため、
  画面が停止せずデモ相当の状態で起動する挙動自体は残っています。

**影響**
- 本番デプロイで環境変数の設定漏れがあると、エラーで気づけず、**実データに接続していないのに一見動いて見える**状態になり得ます。

**対応方針（記録のみ・運用で回避）**
- デプロイ時は必ず環境変数の設定を確認し、デプロイ後の動作確認チェックリスト（[DEPLOYMENT.md](./DEPLOYMENT.md)）で「本番モードで起動していること」を確認してください。
- 本番で環境変数未設定時にハードストップさせたい場合は、`src/lib/supabase.ts` の本番分岐の `try/catch` フォールバックを見直す必要があります（今回のスコープ外）。

## 9. テスト環境の制約（react-scripts 5 と react-router-dom v7 の非互換）

**現状**
- テストランナー（`react-scripts test` = Jest）は、`react-router-dom` v7 の ESM エクスポート構成を解決できず、`src/App.tsx`（`react-router-dom` を import）を描画するテストは**モジュール解決エラーで失敗**します。
- そのため、CRA 標準の雛形テスト `src/App.test.tsx`（`renders learn react link`。本アプリには存在しない要素を検証する無意味なもの）は削除しました。
- 業務ロジックのテスト（`src/utils/workTimeUtils.test.ts` / `overtimeCalculator.test.ts`、計37件）は正常に通過します。

**影響**
- ルーティングを含むコンポーネントの結合テストを Jest（現構成）で書けません。

**対応方針（記録のみ・運用回避）**
- 現状は業務ロジックの単体テストで品質を担保します。
- 将来コンポーネント／E2Eテストが必要な場合は、ビルドツールを Vite + Vitest へ移行するか、Playwright 等の E2E で補うことを検討してください（react-scripts 5 の Jest 設定は CRA がロックしており、`moduleNameMapper` 等の上書きが困難なため）。

## 10. 本番DB実測（2026-07-06）で判明した追加項目

2026-07-06 に本番 Supabase（`pddriyhmkvsklqmtxsro`）を MCP で読み取り、以下を確認しました。
`supabase/schema.sql` は実測値で確定済みです。

### 10-1. 壊れたデッドコード関数（✅ 2026-07-06 削除済み）
- `admin_create_time_record`（uuid版・text版）／`admin_delete_time_record`（uuid版・text版）は、`time_records` に存在しない列（`notes`/`created_by_admin`）参照や型不一致で壊れており、かつフロント未使用のデッドコードでした。
- **対応（実施済み）**: すべて削除しました（`supabase/migrations/0006_drop_broken_admin_time_record_functions.sql`）。打刻修正は `correct_time_record` を使用するため影響ありません。

### 10-2. `updated_at` の自動更新トリガーが未設置（✅ 2026-07-06 設置済み）
- `update_updated_at_column()` 関数は存在するのに、どのテーブルにもトリガーが張られておらず `updated_at` が自動更新されていませんでした。
- **対応（実施済み）**: `employees` / `time_records` / `admin_profiles` に更新トリガーを設置しました（`supabase/migrations/0007_add_updated_at_triggers.sql`）。以後は更新時に `updated_at` が自動反映されます。

### 10-3. 残置バックアップ表 `_recalc_backup_20260606` が RLS 無効で公開状態（✅ 2026-07-06 削除済み）
- 2026-06-06 のステータス再計算時に作られたバックアップ表（1003行）が RLS 無効のまま残存し、匿名ロールから読み取り可能でした（Supabase アドバイザ ERROR `rls_disabled_in_public`）。
- **対応（実施済み）**: 不要と判断し表を削除しました（`supabase/migrations/0005_drop_recalc_backup_20260606.sql`）。アドバイザの ERROR は解消済みです。

### 10-4. Supabase Auth / プラットフォームの推奨設定（アドバイザ WARN）
- 漏洩パスワード保護（HaveIBeenPwned 連携）が**無効**。
- MFA（多要素認証）の選択肢が**不足**。
- Postgres（`supabase-postgres-17.4.1.074`）に**セキュリティパッチ待ち**あり。
- **対応方針（推奨）**: Supabase ダッシュボードで漏洩パスワード保護・MFA を有効化、計画的に Postgres をアップグレード。詳細は [SECURITY.md](./SECURITY.md)。

### 10-5. SECURITY DEFINER 関数が anon から実行可能（アドバイザ WARN）
- `correct_time_record` などの RPC は anon/authenticated から実行可能です（キオスク設計上、打刻・修正に必要）。`admin_*` 系は内部で `is_admin_with_write_access()` により権限チェックしているため、権限のない呼び出しは拒否されます。
- **対応方針（記録のみ）**: 現設計の想定挙動。より厳格にするなら不要な関数の `EXECUTE` 権限を anon から REVOKE することを検討。

## 参考: 今回の整備で解消済みのコード上の問題

- デモ用管理者の認証情報が個人名でソースにハードコードされていた問題は、環境変数化（`REACT_APP_DEMO_ADMIN_USER` / `REACT_APP_DEMO_ADMIN_PASS`、既定 `admin` / `minoru-demo`）により解消しました。
- 本番SupabaseのURLがソースにハードコードされフォールバックしていた問題は除去しました（環境変数必須化）。
- 管理者ログインでパスワードを `localStorage` に平文保存していた問題は撤廃しました（記憶するのはユーザー名のみ）。
- **打刻修正が直行直帰を壊す不具合を修正**しました。従来、`打刻記録管理` の「削除して再作成」は直行直帰フラグを常に false にリセットし、「既存記録を更新」は `is_direct_work` を考慮せずステータス・残業を上書きしていたため、直行直帰の記録を修正すると遅刻/早退/残業が誤付与され、給与時間が不正確になる恐れがありました。共通ヘルパー `applyDirectWorkOverride`（`src/utils/workTimeUtils.ts`）を打刻・修正・再計算の全経路で使用し、直行直帰は「通常扱い・残業0・労働時間は計上」で統一しました（本番DBに該当する破損データが無いことも確認済み）。
